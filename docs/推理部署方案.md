## 一、参考架构

```
机器人本体 (robot client)                    GPU 服务器 (HTTP server)
xRocs/ROS2 控制天工双臂                      FastAPI + openpi policy
┌─────────────────────────┐   HTTP POST    ┌──────────────────────────┐
│ 读3路相机 → JPEG base64  │ ─────────────→ │ 解码图像                  │
│ 读14关节+2夹爪 → state   │  /inference    │ 构造 obs → policy.infer  │
│                         │ ←───────────── │ 返回 action_pred [N,16]  │
│ 阻塞执行前15步 → 再请求   │    JSON        └──────────────────────────┘
└─────────────────────────┘
```

client 执行安全机制（保留不动）：控制周期定频、每步限速 `MAX_DELTA_PER_STEP=0.08 rad`、左右臂关节限位裁剪、夹爪 clip 到 [0,1]、启动时先移动到数据集首帧均值位姿、`CHUNK_CONFIRM` 逐块确认开关。

## 二、布局方案：全链路统一训练布局

**决策：整条链路（client 采集 → HTTP → server → 模型 → HTTP → client 执行）统一使用训练布局，server 不做任何重排，降低长期维护风险；代价是真机执行索引必须同步修改（见第四节索引清单）。**

训练布局（`TienkungRepack`，`src/openpi/training/config.py`）：

| 索引 | 内容 |
| --- | --- |
| 0:7 | 左臂 7 关节（rad） |
| 7:14 | 右臂 7 关节（rad） |
| 14 | 左夹爪开合 [0,1] |
| 15 | 右夹爪开合 [0,1] |

对比：`参考/client.py` 的旧布局是 `[左臂7, 左爪1, 右臂7, 右爪1]`（`state[7]`=左爪、`state[8:15]`=右臂），**新 client 所有索引必须按训练布局重写，不能照抄**。

数据流约定：

- client `_get_state_for_server()` 输出 `[arm[:7], arm[7:14], left_hand, right_hand]`（即训练布局）
- server 直接把收到的 state 放入 `observation/state`，不重排
- server 返回的 `action_pred` 保持训练布局 `[L7, R7, Lg, Rg]`
- client 执行动作按新索引：左臂 `0:7`，右臂 `7:14`，左夹爪 `14`，右夹爪 `15`

### server 侧观测键（对照 `src/openpi/policies/tienkung_policy.py` 的 `TienkungInputs`）

| 键 | 内容 |
| --- | --- |
| `observation/state` | [16] float，训练布局，透传 |
| `observation/images/camera_top` | uint8 HWC RGB（任意分辨率，`ResizeImages(224,224)` 与训练一致） |
| `observation/images/camera_left` | 同上 |
| `observation/images/camera_right` | 同上 |
| `prompt` | 任务文本 |

已核验：JPEG 解码出的 uint8 HWC 图像经 `_parse_image` 是 no-op；`PadStatesAndActions` 把 16 维 pad 到 32，与训练一致。

### client 侧数据集绑定常量

| 常量 | 旧值（参考代码） | 本任务值 | 依据 |
| --- | --- | --- | --- |
| `COLLECTION_INIT_POSE` | 旧任务首帧均值（旧布局） | 见下，**新布局** | 401 条 episode 首帧均值，std ≤ 0.12 rad |
| 默认 `--task` | `ur2_place_eggplant_to_plate` | 训练 prompt（见下方代码块） | 必须与 `meta/tasks.jsonl` **逐字一致** |
| `CONTROL_DT` | 1/25 | **1/29** | 数据集 fps=29 |
| `CHUNK_EXEC_STEPS` | 15 | 15（保留） | horizon 32 执行约一半后重新推理 |

训练 prompt（逐字，来自 `meta/tasks.jsonl`）：

```
Use an electrostatic roller to pick up debris on the desktop
```

新 `COLLECTION_INIT_POSE`（**训练布局 [L7, R7, Lg, Rg]**，由数据集实算）：

## 三、已核验无需改动的部分

- **delta 动作还原**：`create_trained_policy` 输出端自动做 Unnormalize → `AbsoluteActions`（基于请求携带的当前 state）→ `TienkungOutputs` 截断回 16 维，返回**绝对关节角（rad）+ 夹爪绝对开合**。离线评测 `scripts/eval_offline.py` 已实证此链路正确（29999 步右臂 MAE≈0.0075 rad）。
- **norm stats**：从 `<checkpoint>/assets/tienkung_lint_roller_lerobot/norm_stats.json` 加载（`create_trained_policy` 默认逻辑），与训练严格一致；v1 权重左臂会被旧 stats 钉在静止区间，部署安全。
- **HTTP 协议字段**：`action_pred` / `request_id` / `policy_timing` / `server_timing` 保留，协议不变。

## 四、实施步骤

### 1. GPU 侧：`scripts/deploy/tienkung/serve_http.py`

以 `参考/server.py` 为骨架：

- startup 时 `create_trained_policy(cfg, checkpoint_dir)`（norm stats 默认从 checkpoint assets 读，保留 `--assets-dir` 覆盖入口）
- `/inference`：base64 解码 3 路图 → 组装观测（state 透传，不重排）→ `policy.infer` → 返回动作块（不重排）
- 启动参数：`--config-name / --checkpoint-dir / --assets-dir / --host / --port`

**依赖（已完成）**：`fastapi`、`uvicorn`、`httpx` 均已 `uv add` 进 pyproject.toml 并实测可从 `.venv` import（httpx 供 GPU 机上直接调试 client HTTP 部分；机器人侧仍用旧环境自带的 httpx）。

### 2. 机器人侧：`scripts/deploy/tienkung/robot_client.py`

复制 `参考/client.py` 后修改。**所有旧布局索引位置清单（实现时逐处核对，一处都不能漏）**：

| 位置 | 旧代码（旧布局） | 新代码（训练布局） |
| --- | --- | --- |
| `_get_state_for_server` | `concat([arm[:7], [left_hand], arm[7:14], [right_hand]])` | `concat([arm[:14], [left_hand], [right_hand]])` |
| `_clip_action` 右臂限位 | `action[8 + i]` | `action[7 + i]` |
| `_clip_action` 夹爪 | `action[7]`, `action[15]` | `action[14]`, `action[15]` |
| `_command_action` 手臂目标 | `concat([q[:7], q[8:15]])` | `q[:14]` |
| `_command_action` 夹爪下发 | `q[7]`, `q[15]` | `q[14]`, `q[15]` |
| `prepare` 手臂目标 | `concat([POSE[:7], POSE[8:15]])` | `POSE[:14]` |
| `prepare` 夹爪 | `POSE[7]`, `POSE[15]` | `POSE[14]`, `POSE[15]` |
| 日志打印 | `L0=q[0] Lg=q[7] R0=q[8] Rg=q[15]` | `L0=q[0] R0=q[7] Lg=q[14] Rg=q[15]` |

同时更新第二节表格中的常量（`COLLECTION_INIT_POSE` 新布局、默认 task、`CONTROL_DT=1/29`）。

机器人本体用旧项目 Python 环境运行（xRocs/httpx/cv2 已具备），不依赖本仓库 venv。

### 3. 项目 `.venv` 迁移要求（GPU 侧）

用户目标：把 openpi 推理需要的包都放到 `/mnt/cpk/magic/openpi/.venv` 下，项目整体迁移到同类 GPU 机器时尽量不重新配环境。

当前实测状态：

| 项 | 结果 |
| --- | --- |
| `.venv` 路径 | `/mnt/cpk/magic/openpi/.venv` |
| Python | 3.11.11 |
| `.venv/bin/python` | 绝对软链接到 `/usr/local/bin/python3.11` |
| 已有关键包 | `jax==0.5.3`、`jaxlib==0.5.3`、`torch==2.7.1+cu126`、`cv2`、`lerobot`、`openpi`、`openpi_client`、`fastapi`、`uvicorn`、`httpx` |
| 安装方式 | `openpi` / `openpi-client` 为 editable install，依赖仓库源码路径 |
| 体积 | `.venv` 约 7.8G；29999 checkpoint 约 42G |

迁移约束：

- 目标机推荐使用相同或高度一致的 Linux / glibc / Python 3.11 / CUDA 驱动环境。
- 项目路径固定为 `/mnt/cpk/magic/openpi`；否则 editable 安装、脚本 shebang 和部分绝对路径需要修复。
- 目标机必须存在 `/usr/local/bin/python3.11`，或迁移后重建 `.venv/bin/python` 软链接。
- `.venv` 只覆盖 GPU 服务器侧 openpi 推理环境；机器人端 xRocs/ROS2 仍使用机器人旧环境。
- 不保证跨系统、跨 Python 小版本、跨 CUDA/驱动大版本时复制 `.venv` 可直接运行。

### 4. 本地验证（无机器人，GPU 机上完成）

用 `scripts/deploy/tienkung/verify_serve_http.py`（带硬断言，任一失败非零退出；内置 `HF_LEROBOT_HOME` 默认值，离线可跑）：

```bash
# server 起在 18000（与部署命令一致）后：
JAX_PLATFORMS=cpu uv run python scripts/deploy/tienkung/verify_serve_http.py
# 可选参数：--server / --episode / --config-name / --exp-name / --step
```

断言项：chunk 形状 `[32,16]`、右臂 MAE < 0.05 rad、左臂 MAE < 0.01 rad（钉在静止值）、右臂运动幅度与真值同量级（防布局错位）、首步动作与 state 偏差 < 0.05 rad、夹爪落在 [-0.1, 1.1]、与 eval_offline 同帧误差同量级。

**已实测（2026-07-03，29999 checkpoint，全部 PASS）**：右臂 MAE 0.037、左臂 MAE 0.0005、运动幅度比 1.05、首步偏差 0.0015 rad；稳态延迟模型 32ms / RTT 69ms（本机回环），首次请求含 JIT 编译约 18s（client 60s 超时可容忍）；错端口时报错清晰并以非零码退出。

### 5. 真机前 dry-run（`--dry-run`，不向机器人下发任何指令）

- `--dry-run` 语义：不向机器人下发任何指令，包括运动目标、夹爪指令和 `set_dt/set_gain/set_lookahead_time` 控制器参数设置（全部打印代替）
- 打印 state 首帧与 action 首步的全部 16 维及分段标注，人工核对左/右臂、夹爪落位
- 确认 `_command_action` 的 arm target 为 `q[:14]`，夹爪取 `q[14]`/`q[15]`
- 核对 `_get_state_for_server()` 读数与 `COLLECTION_INIT_POSE` 接近（prepare 已内置自动打印最大偏差；同时验证真机关节顺序与数据集 puppet 字段一致——同为 xRocs 栈预期一致，需实测一次）

### 6. 真机首跑检查单

1. server 用 `29999` checkpoint 启动；client `--task` 逐字用训练 prompt
2. `CHUNK_CONFIRM=True` 逐块确认模式起步
3. 观察左臂：预期基本静止（被 norm stats 钉住）；若左臂有异常动作立即停
4. 确认相机键名含 top/left/right（`_pick_image_key` 模糊匹配）

## 六、风险与备注

| 风险 | 说明与对策 |
| --- | --- |
| client 索引改漏 | 全链路统一布局后风险集中在 client 改写。第四节清单逐处核对 + dry-run 打印分段核验 |
| 真机关节顺序与数据集不一致 | dry-run 核对读数 vs `COLLECTION_INIT_POSE` |
| prompt 不一致 | tokenize 后条件分布偏移，效果无声下降。逐字复制 `meta/tasks.jsonl` |
| 控制频率失配 | 数据 29fps vs 参考 25Hz。`CONTROL_DT=1/29`；xRocs 伺服参数 `set_dt` 同步改 |
| JPEG 压缩损失 | 训练用无损视频解码帧，推理 quality=90 JPEG，预期影响可忽略；效果异常可改 PNG 排查 |
| 推理延迟 | blocking 模式执行 15 步（约 0.52s）后停下等推理，延迟高会卡顿；后续可考虑 openpi 自带 `ActionChunkBroker` 异步方案 |
| 依赖不可复现 | 已解决：fastapi / uvicorn / httpx 均已 `uv add` 进 pyproject.toml 且 `.venv` 可 import |
| `.venv` 迁移失败 | 当前 `.venv/bin/python` 指向 `/usr/local/bin/python3.11`，且 openpi 为 editable install。目标机需同路径、同类系统和 Python/CUDA 环境 |
